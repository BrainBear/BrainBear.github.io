Handler 是什么

Handler 是 Android 中最常用的消息机制，主要用于异步消息的处理。

Handler 的工作原理

Handler 的核心类有：

  • Handler：用户主要操作的类，主要用于发送消息，然后处理消息
  • Looper：是整个功能的核心，一个线程只有一个 Looper 对象，它内部维护了一个消息队列，对外提供添加消息的接口，还有就是分发消息
  • MessageQueue：消息队列
  • Message：消息主体

我们先看看 IdleHandler 的定义：

从注释中可以看到，当消息队列中没有需要执行的消息时，则会回调这个接口。需要注意的是：即使队列中有消息，但是还没有到对应的执行时间的话，也是会调用的。

IdleHandler 也是在 MessageQueue 读取消息的时候生效：

那么这个 IdleHandler 有什么用呢？因为它回调的时候证明当前消息队列没有别的需要执行的任务，这样可以去做一下不需要马上执行的任务，而等到相对比较空间的时候再执行，相当于一种性能优化。

为什么一直阻塞但是不会 ANR ？

Android 系统的 ANR 检测其实也是通过 Handler 来处理的，它是发送了一个延时的回调来检测的，如果在延时执行前取消这个回调则相关无事，否则就会显示 ANR 。所以虽然 Looper 一直阻塞，但是发送消息的时候会主动唤醒去执行，所以正常情况下是不会 ANR 的。但是如果是在主线程中执行了一个耗时的操作，这样会阻塞消息队列的消息，导致没有及时取消 ANR 检测，这样 ANR 就发生了。

Binder 是什么

Binder 是 Android 提供的一种通信机制，主要用于跨进程通信。

为什么要用 Binder

Android 是基于 Linux 的系统，Linux 已经有很多种跨进程通信的机制，例如：socket、管道、共享内存等。

Binder 共享内存 socket
性能 数据复制一次 无需复制 需要复制两次
易用性 基于 C/S 架构，面向对象 控制复杂，需要各种同步逻辑 通用接口
安全性 由系统保证通信安全 依赖上层协议 依赖上层协议

由上面的表格对比可以知道，Binder 是基于性能、易用性和安全性综合考虑的结果。
