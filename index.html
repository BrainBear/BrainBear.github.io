JVM：内存模型

Posted on 2019-12-14

内存模型是什么

因为 Java 虚拟机是在硬件之上抽象的一层，Java 为了实现”一次编写，到处运行“的效果，在 Java 虚拟机规范中定义了一个 Java 内存模型（Java Memory Model, JMM）来屏蔽各种硬件和操作系统的内存访问差异，以实现让 Java 程序在各种平台下都等达到一致的内存访问效果。

Java 内存模型主要有三个内容：有序性、可见性和原子性。

为什么会有可见性问题

Java 内存模型规定了所有的变量都存储在主内存中，但是每个线程还有自己的工作内存，线程的工作内存中保存了被该线程使用到的变量的主内存副本拷贝。线程对变量的所有操作（读取、赋值）都必须在工作内存中进行，而不能直接读写主内存的变量，不同的线程之间也无法直接访问对方工作内存的变量，线程间变量值的传递需要通过主内存来完成。

由于工作内存和主内存的同步时间不确定 ，所以就会有可见性的问题。

Happens-Before

在 Java 语言中有一个”先行发生“（happens-before）原则，它是判断数据是否存在竞争、线程是否安全的主要依据。我们可以通过这个原则解决并发环境下两个操作之间是否可能存在冲突的所有问题。

Happens-Before 是 Java 内存模型中定义的两项操作之间的偏序关系，如果说操作 A 先行发生于操作 B，其实就是说在发生操作 B 之前，操作 A 产生的影响能被操作 B 观察到，”影响“包括修改了内存中共享变量的值、发送了消息、调用了方法等。

Happens-Before 有几个规则：

  • 程序次序规则
    在一个想成内，按照程序代码顺序，书写在前面的操作先行发生与书写在后面的操作。

  • 管程锁定规则
    一个解锁操作先行发生于后面对同一个锁的加锁操作。这里的后面是指时间。简单理解就是对于同一个锁，加锁之后的内容可以看到解锁之前的全部操作。

  • volatile 变量规则
    对一个 volatile 变量的写操作先行发生于后面对这个变量的读操作。这里的后面也是指的时间。

  • 线程启动规则
    Thread 对象的 start() 方法先行发生于此线程的每个动作。

  • 线程终止规则
    线程中的所有操作都先行发生于对此线程的终止检测。

  • 线程中断规则
    对线程interrupt()方法的调用先行发生与被中断线程的代码检测到中断事件的发生。

  • 对象终结规则
    一个对象的初始化完成（构造函数执行结束）先行发生于它的 finalize() 方法的开始

  • 传递性
    如果操作 A 先行发生于操作 B，操作 B 先行发生于操作 C，那么操作 A 先行发生于操作 C。

原子性

如果一个（或一系列）操作，要么全部执行成功，要么全部不执行，不会出现执行一半的情况，那么可以称这个操作是原子性的。

在 Java 中基本类型的赋值操作（Java 虚拟机规范对 long 和 double 类型的赋值没有强制要求原子性，但是商用虚拟机基本都做了原子保证）、引用的 reference 赋值操作都是原子性的，而 Atomic 包也提供了很多原子性的类。

volatile

volatile 是 Java 中的一个关键字，是一种轻量级的同步机制，但是 volatile 做不到锁机制的原子保护，volatile 的只有两个作用：

  • 保证变量对所有线程的可见性
  • 禁止指令重排序

上面演示重排序和可见性的代码，都可以通过给变量增加 volatile 修饰来解决。

但是 volatile 并不能保证原子性，如果对一个 volatile 修饰的整形变量在多线程环境下自增，最后的结果肯定是会错误的。

JVM：运行时数据区

Posted on 2019-12-01

运行时数据区

JVM 管理的内存分为几个区域：

程序计数器

程序计数器是一块比较小的内存空间，如果线程在执行 Java 方法，那么计数器存储的是正在执行的虚拟机字节码指定的地址，如果正在执行的是 Native 方法，计数器的值为 Undefined 。

字节码解释器通过计数器的值来读取下一条需要执行的字节码指令。因为 Java 支持多线程，为了线程切换后能恢复到正确的执行位置，所以每条线程都需要有一个独立的程序计数器，每个计数器记录当前线程的执行置位，不同的线程互不干扰，所以程序计数器是线程私有的。

程序计数器随着线程的创建而创建，线程结束的时候销毁。它是唯一一个在 Java 虚拟机规范中没有规定任何 OutOfMemoryError 的区域。

Java 虚拟机栈

Java 虚拟机栈是描述 Java 方法运行过程的内存模型。每个方法在执行的时候会创建一个栈帧（Stack Frame），方法从调用到执行完成的过程，就对应栈帧在虚拟机栈中的入栈和出栈。栈帧存储方法在执行过程中的一些信息：

  • 局部变量表
  • 操作数栈
  • 动态链接
  • 方法出口信息
  • …

在 Java 虚拟机规范中，虚拟机栈可以设置固定的深度，也可以动态扩展。如果是固定深度，当线程请求的栈深度超过设置的深度，JVM 会抛出 StackOverFlowError ；如果虚拟机栈允许动态扩展，当拓展的时候无法申请到足够的内存，JVM 会抛出 OutOfMemoryError 异常。

局部变量表

局部变量表用于存放方法参数和方法执行中产生的局部变量，它的大小在编译的时候就可以确定。局部变量表的容量以变量槽（Variable Slot）为单位，大小为 32 位，如果要存储 64 位的类型数据（如 long 和 double）会占用两个 slot。

在实例方法中，局部变量表第 0 位默认存储方法所属对象实例的引用，所以可以在实例方法中通过 this 来访问当前对象。如果是静态方法则没有这个引用。

操作数栈

操作数栈用于存储方法执行中各个指令操作的数据，包括方法的返回值（如果方法有返回值的话）。

动态链接

每个栈帧持有一个指向运行时常量池中该栈帧所属方法的引用，持有这个引用是为了支持方法调用过程中的动态链接。

这块主要和方法的调用有关，但是查阅了很多资料也是看的有点懵，先不管了。

方法返回地址

对于方法执行之后，无论是正常执行完，还是发生异常退出。都需要返回到方法被调用的位置，方法返回地址就存储了这个信息。

本地方法栈

本地方法栈 (native method stack) 的作用和 Java 虚拟机栈非常类似，气不过本地方法栈是为了本地方法服务的。JVM 规范中对本地方法栈中使用什么语言，使用方式和数据结构都没有强制规定，不同的虚拟机可以有不同的实现。

堆

堆用于存放对象实例。堆是线程共享的。所以会有线程同步的问题。

堆在 JVM 启动的时候就创建了，根据 Java 虚拟机规范的规定，堆可以使用物理上不连续的内存空间，只要逻辑上是连续的就可以。

堆也是垃圾回收的主要场所。现代的垃圾回收基本都使用分代收集的算法，可以把堆分为新生代（Eden、From Survivor、To Survivor）和老年代。

方法区

方法区和 Java 堆一样是线程共享的。它主要用于存储以下信息：

  • 已被虚拟机加载的类信息
  • 常量
  • 静态变量
  • 几时编译器编译后的代码
  • …

方法区也有可能垃圾回收，主要的目标是针对常量池的回收和对类型的卸载，但是对这些的回收效益比较低，所以 Java 虚拟机规范中不强制要求对方法区垃圾回收。

运行时常量池是方法区的一部分。当类被加载进虚拟机后，类中的常量（如编译期生成的字面量和符号引用）就存储在常量池中。

多线程：CAS

Posted on 2019-11-07

CAS

CAS(compare and swap) 是一种算法，意思是比较再交换。CAS 有三个操作数：内存值 V、预期值 A、要修改的值 B，CAS 可以描述为当且仅当内存值 V 和预期值 A 相等，才把内存值修改为 B。否则什么都不做，一般还会返回当前的内存值 V。

假设有两个线程同时去更新 value 的值，一个加一，一个减一，thread1 和 thread2 的预期值都是当前内存值 56，假设 thread1 先执行，通过 cas 比较了预期值和内存值都是 56，所以成功把 value 更新为 57，但是 thread2 去执行 cas 的时候发现内存值 57 和预期值 56 不一致，所以更新失败，但是 thread2 可以重试。

CAS 是由 CPU 来保证比较和更新操作的原子性的，所以不会有线程安全的问题。

多线程：Lock

Posted on 2019-11-06

为什么需要 Lock

Lock 也是为了保证并发安全的措施，那么 Java 已经提供 synchronized 关键字了，为什么还需要 Lock 呢？因为 synchronized 有以下几个缺点：

  • 获取锁的时候不能设置超时
  • 获取锁的时候不能被终端
  • 加锁和释放的逻辑单一，不够灵活
  • 无法知道是否成功获取到锁

因为解决以上的问题，Lock 提供了几个接口：

lock()

lock() 就是最普通的获取锁，如果锁已经被其他线程获取了，那么当前线程会等待。和 synchronized 不一样的是，如果在临界区发生异常，Lock 不会自动释放锁，需要手动释放。

最佳实践就是在 finally 语句中手动释放锁。

和 synchronized 类似，lock() 也不能被中断，想要响应中断需要使用 lockInterruptibly()

lockInterruptibly()

在 lockInterruptibly() 获取锁的等待期间，如果线程被中断，会抛出 InterruptedException 异常。

tryLock()

tryLock() 和 lock() 最大的区别是马上返回的，它会尝试去获取锁，如果获取到了，就返回 true ，如果锁被其他线程持有，它会马上返回 false 。这在一些临界区有多个锁的时候可以方便地避免死锁。

tryLock(long time, TimeUnit unit) 则可以等待指定的时间，如果在指定时间内无法获取到锁则返回 false ，而且在等待期间是可以响应中断的。

锁分类

锁具有很多特点，按照不同的特点来分类可以分很多种类

悲观锁&乐观锁

从是否锁住资源来分可以分为悲观锁和乐观锁，悲观锁也称为互斥同步锁，乐观锁也称为非互斥同步锁。

synchronized 和 Lock 都是悲观锁，那么为什么需要乐观锁呢？

  1. 互斥造成的阻塞和唤醒会带来性能上的消耗
  2. 互斥在操作不当的时候有可能造成死锁，这是非常严重的问题
  3. 有可能低优先级的线程获取到了锁，导致高优先级的线程等待，这个称为优先级反转

而乐观锁是默认在操作过程中没有其他线程来干扰，在操作完要更新的时候，会去比对操作过程中是否有被其他线程修改多，如果没有就直接更新。如果有被修改过再继续重试或者报错。乐观锁一般都是基于CAS来实现的。最常用的例子就是原子类。

悲观锁的原始开销是要高于乐观锁的，但是在竞争非常激烈的情况下，如果临界区大，乐观锁的多个线程会长时间自旋，对 CPU 的消耗就会比较大。所以乐观锁一般都是运用在临界区比较小的地方。

公平锁&非公平锁

公平锁是指按照线程请求的顺序来分配锁，而非公平锁是不完全按照请求的顺序，在一定条件下可以插队。Lock 默认都是非公平的，因为非公平可以提交效率，例如一个线程释放了锁，准备去唤醒其他线程的时候，如果这个时候有另外一个线程来请求锁，那么直接把锁分配给它，可以节省唤醒的消耗。

共享锁&独占锁

独占锁是指一个线程获取到了锁之后，其他线程就无法访问对应的资源了，但是共享锁不一样，一个线程获取到锁之后，其他线程也可以访问对应的资源，可以查看但是无法修改或删除数据，所以也称为读锁。

JDK 提供了 ReentrantReadWriteLock 读写锁，其中读锁是共享的，写锁是独占的。因为在多线程并发读取是不会造成线程安全问题的，只有写操作才会有问题。如果是在读的情况比写的情况要多的话，使用读写锁可以提高效率。

读写锁可以同一时间上有多个线程获取到读锁，但是最多只有一个线程获取到写锁，不会存在多个线程同时读写的情况。

读写锁有一个升降级的问题，线程在获取到写锁的条件下，直接去获取读锁，称为锁的降级。如果线程在获取到读锁的条件下，直接去获取写锁，称为锁的升级。ReentrantReadWriteLock是支持降级不支持升级的。下面我们来演示一下：

多线程：线程池

Posted on 2019-11-05

线程池

在实际项目中，我们经常使用线程，但是如果每次使用的时候都去创建一个新的线程，会给 CPU 和内存造成比较大的开销。而 Java 提供了线程池来解决这个问题。使用线程池可以加快异步的响应速度，更加合理地利用 CPU 和内存，并且可以很方便地管理一系列的线程。

创建

线程池的构造方法：

参数名 类型 含义
corePoolSize int 核心线程数
maximumPoolSize int 最大线程数
keepAliveTime long 保持存活时间
unit TimeUnit 存活时间单位
workQueue BlockingQueue 任务队列
threadFactory ThreadFactory 线程工厂，用来创建新的线程
handler RejectedExecutionHandler 当线程池无法接收任务时的拒绝策略

在线程池中，核心线程是不会被回收的，而普通的线程在超过了存活时间还没有任务要执行的话，它就会被回收掉。

创建新线程的时机

当提交了一个任务之后：

  1. 如果当前线程池的线程数量少于 corePoolSize，会创建一个新的线程
  2. 如果线程数大于等于 corePoolSize，但是少于 maximumPoolSize，则会将任务放入到任务队列中
  3. 如果任务队列满了，并且线程数少于 maximumPoolSize ，则会创建一个新的线程
  4. 如果任务队列满了，并且线程数大于 maximumPoolSize ，则会拒绝这个任务

如果有特殊的需求，例如自定义线程名字，优先级等等，可以自己实现一个 ThreadFactory 。

任务队列

任务队列分两种：

  • 无界队列
    • LinkedBlockingQueue
      如果使用了无界队列，那么线程池的线程数永远都不会超过 corePoolSize，后续的任务全部会存储到队列中
  • 有界队列
    • ArrayBlockingQueue
    • SynchronousQueue
      ArrayBlockingQueue 可以指定队列的大小，但是 SynchronousQueue 不可以，它的大小永远为 0

常见的线程池

JDK 提供了几种类型的线程池：

FixedThreadPool

FixedThreadPool 指定了核心线程数和最大线程数相等，而且任务队列是无界的 LinkedBlockingQueue，所以当任务处理比较慢，又一直提交新任务的时候，未处理的任务就会堆积在任务队列中，严重时可能会造成 OOM。

SingleThreadExecutor

SingleThreadExecutor 顾名思义，它就只有一个线程，相当于指定线程数为 1 的 FixedThreadPool。

CachedThreadPool

CachedThreadPool 的核心线程数为 0 ，但是最大线程数是 Integer.MAX_VALUE ，而且它的任务队列是 SynchronousQueue，由于 SynchronousQueue 的容量固定为 0 ，所以当你提交任务的时候，如果当前没有空闲的线程，它就会创建一个新的线程来执行。在极端情况下会创建非常多的线程，从而造成 OOM。

ScheduledThreadPool

ScheduledThreadPool 和其他线程池最不一样的地方是它提供了任务调度的接口，可以实现定时、周期执行任务。具体的接口就不在这里列举了。

停止线程池

线程池提供了 shutdown() 方法来停止线程，但是这个停止是安全的，它会标识线程池已经进入停止状态了，但是它会把任务队列中的任务全部执行完才会真正停止线程池。但是调用 shutdown() 方法之后，用户就无法提交新的任务了，提交会直接拒绝。

线程池提供了 isShutdown() 方法来判断是否停止，如果用户调用了 shutdown() 方法，哪怕还有任务未执行完，isShutdown() 方法也是会返回 true，如果想获取真正的停止状态，需要调用 isTerminated() 方法。

线程池也提供了一个 shutdownNow() 方法来马上停止线程池，它会网线程池中的全部线程发送中断信号，而且会返回任务队列中全部未执行的任务。

拒绝任务

当线程池关闭，或者已经达到最大线程数并且任务队列已满时，如果提交新的任务，线程池会拒绝。JDK 默认提供了以下拒绝策略：

策略 处理逻辑
AbortPolicy 丢弃任务并抛出RejectedExecutionException异常
DiscardPolicy 丢弃任务，但是不抛出异常
DiscardOldestPolicy 丢弃队列最前面的任务，然后重新提交被拒绝的任务
CallerRunsPolicy 由调用线程（提交任务的线程）处理该任务
