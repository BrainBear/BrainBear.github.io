Android UI 刷新原理

Posted on 2020-09-01

前言

众所周知，手机的显示帧率要到达 60FPS 才达到流畅的标准，现在（2020 年 9 月）移动设备最高已经有 120 FPS 刷新率。那么 Android 内部是怎么协调 UI 的刷新的呢？

从 invalidate 开始

从 invalidate 开始，一直跟踪源码可以发现以下调用逻辑：

sequenceDiagram
    autonumber
    View->>ViewGroup: invalidateInternal
    loop 一直向父容器调用
        ViewGroup->>ViewGroup: onDescendantInvalidated
    end
    ViewGroup->>ViewRootImpl: onDescendantInvalidated
    ViewRootImpl->>Choreographer: postCallback
    Choreographer->>ViewRootImpl: 回调

可以看到 ViewRootImpl 往 choreographer 注册了一个回调，所以重点去看 ViewRootImpl 和 choreographer 的代码

ViewRootImpl

ViewRootImpl 是 View 树结构中的根节点，以后有机会再详细了解它，今天我们只看和刷新相关的逻辑：

  1. 在入口处使用 mTraversalScheduled 字段来防止重入，也就是说哪怕在一帧时间内，子控件多次触发重绘，也会在这个方法中被拦截，最终只会有一个回调任务设置到 Choreographer 中
  2. 在注册回调前往当前线程的 Handler 中设置了消息屏障，为什么要设置屏障呢？因为有可能线程任务比较重，但是界面刷新的优先级肯定是要比其他任务更重要的，设置了消息屏障之后，会让 Handler 优先去处理异步消息
  3. 往 Choreographer 中插入了一个 CALLBACK_TRAVERSAL 类型的任务 mTraversalRunnable

Choreographer

从上面的逻辑来看，ViewRootImpl 只是根据 Choreographer 的回调来处理刷新的逻辑，所以帧率的控制肯定是在 Choreographer 内部。

任务回调

在 Choreographer 中定义了五种任务类型：

Choreographer 通过 postCallback 来插入任务，在做了一些参数的安全校验之后调用到了 postCallbackDelayedInternal 方法：

  1. 根据任务类型放入不同的队列中
  2. 如果没有延时则会直接调用 scheduleFrameLocked
  3. 有延迟则通过 Handler 发送异步消息处理，但是最终也是会走到 scheduleFrameLocked

这里的 mHandler 是 FrameHandler 的实例

在 doScheduleCallback 中也是调用了 doScheduleCallback 方法

申请 vsync

mDisplayEventReceiver 是 FrameDisplayEventReceiver 的实例，而 FrameDisplayEventReceiver 继承于抽象类 DisplayEventReceiver ，DisplayEventReceiver 的代码逻辑比较简单，核心都在 native 层，这里就不详细分析了，简单来说就是通过 scheduleVsync 申请 vsync 信号，然后 native 层会回调到 dispatchVsync 方法，最终调到了 FrameDisplayEventReceiver 的 onVsync

在 doFrame 中调用 doCallbacks 来回调任务：

整体的流程如下：

graph TD
    postCallbackDelayedInternal -->|没有延时直接调用| scheduleFrameLocked
    postCallbackDelayedInternal -->|有延时则通过 mHandler调用| scheduleFrameLocked
    scheduleFrameLocked --> Vsync{是否使用 Vsync}
    Vsync -->|N| doFrame
    Vsync -->|Y| threadCheck{当前线程是否是Choreographer绑定的线程}
    threadCheck -->|Y 直接调用| scheduleVsyncLocked
    threadCheck -->|N 通过 mHandler 调用| scheduleVsyncLocked
    scheduleVsyncLocked --> displayEvent(申请 Vsync 信号)
    displayEvent --> onVsync(onVsync)
    onVsync --> doFrame
    doFrame --> doCallbacks
