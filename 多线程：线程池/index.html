多线程：线程池

Posted on 2019-11-05

线程池

在实际项目中，我们经常使用线程，但是如果每次使用的时候都去创建一个新的线程，会给 CPU 和内存造成比较大的开销。而 Java 提供了线程池来解决这个问题。使用线程池可以加快异步的响应速度，更加合理地利用 CPU 和内存，并且可以很方便地管理一系列的线程。

创建

线程池的构造方法：

参数名 类型 含义
corePoolSize int 核心线程数
maximumPoolSize int 最大线程数
keepAliveTime long 保持存活时间
unit TimeUnit 存活时间单位
workQueue BlockingQueue 任务队列
threadFactory ThreadFactory 线程工厂，用来创建新的线程
handler RejectedExecutionHandler 当线程池无法接收任务时的拒绝策略

在线程池中，核心线程是不会被回收的，而普通的线程在超过了存活时间还没有任务要执行的话，它就会被回收掉。

创建新线程的时机

当提交了一个任务之后：

  1. 如果当前线程池的线程数量少于 corePoolSize，会创建一个新的线程
  2. 如果线程数大于等于 corePoolSize，但是少于 maximumPoolSize，则会将任务放入到任务队列中
  3. 如果任务队列满了，并且线程数少于 maximumPoolSize ，则会创建一个新的线程
  4. 如果任务队列满了，并且线程数大于 maximumPoolSize ，则会拒绝这个任务

ThreadFactory

线程池的线程都是通过 ThreadFactory 来创建的，默认已经提供了一个线程工厂：

如果有特殊的需求，例如自定义线程名字，优先级等等，可以自己实现一个 ThreadFactory 。

任务队列

任务队列分两种：

  • 无界队列
    • LinkedBlockingQueue
      如果使用了无界队列，那么线程池的线程数永远都不会超过 corePoolSize，后续的任务全部会存储到队列中
  • 有界队列
    • ArrayBlockingQueue
    • SynchronousQueue
      ArrayBlockingQueue 可以指定队列的大小，但是 SynchronousQueue 不可以，它的大小永远为 0

常见的线程池

JDK 提供了几种类型的线程池：

FixedThreadPool

FixedThreadPool 指定了核心线程数和最大线程数相等，而且任务队列是无界的 LinkedBlockingQueue，所以当任务处理比较慢，又一直提交新任务的时候，未处理的任务就会堆积在任务队列中，严重时可能会造成 OOM。

SingleThreadExecutor

SingleThreadExecutor 顾名思义，它就只有一个线程，相当于指定线程数为 1 的 FixedThreadPool。

CachedThreadPool

CachedThreadPool 的核心线程数为 0 ，但是最大线程数是 Integer.MAX_VALUE ，而且它的任务队列是 SynchronousQueue，由于 SynchronousQueue 的容量固定为 0 ，所以当你提交任务的时候，如果当前没有空闲的线程，它就会创建一个新的线程来执行。在极端情况下会创建非常多的线程，从而造成 OOM。

ScheduledThreadPool

ScheduledThreadPool 和其他线程池最不一样的地方是它提供了任务调度的接口，可以实现定时、周期执行任务。具体的接口就不在这里列举了。

停止线程池

线程池提供了 shutdown() 方法来停止线程，但是这个停止是安全的，它会标识线程池已经进入停止状态了，但是它会把任务队列中的任务全部执行完才会真正停止线程池。但是调用 shutdown() 方法之后，用户就无法提交新的任务了，提交会直接拒绝。

线程池提供了 isShutdown() 方法来判断是否停止，如果用户调用了 shutdown() 方法，哪怕还有任务未执行完，isShutdown() 方法也是会返回 true，如果想获取真正的停止状态，需要调用 isTerminated() 方法。

线程池也提供了一个 shutdownNow() 方法来马上停止线程池，它会网线程池中的全部线程发送中断信号，而且会返回任务队列中全部未执行的任务。

拒绝任务

当线程池关闭，或者已经达到最大线程数并且任务队列已满时，如果提交新的任务，线程池会拒绝。JDK 默认提供了以下拒绝策略：

策略 处理逻辑
AbortPolicy 丢弃任务并抛出RejectedExecutionException异常
DiscardPolicy 丢弃任务，但是不抛出异常
DiscardOldestPolicy 丢弃队列最前面的任务，然后重新提交被拒绝的任务
CallerRunsPolicy 由调用线程（提交任务的线程）处理该任务
