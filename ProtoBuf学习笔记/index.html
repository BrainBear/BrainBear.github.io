ProtoBuf学习笔记

Posted on 2018-08-19

Protocol Buffers 是Google开源的一种跨平台、多语音的数据序列化协议。其实之前就一直在使用，只是拖到了今天才打算写个笔记记录一下。

参考

高效的数据压缩编码方式 Protobuf
官方文档

优缺点

ProtoBuf 对比当前热门的 JSON 和 XML 格式，JSON 和 XML 都是通过字符串存储传输的，而 ProtoBuf 是将数据编码成了二进制格式存储传输。这是它的优点，占用的内存小，序列化和反序列化的速度快。但是这也是它的缺点，因为编码成了二进制会让数据无法像JSON和XML那样直观，不方便debug。

准备工作

首先需要准备编译环境，要到 Github 下载 protoc 程序。根据自己的系统和版本号下载就可以了，我下载的是 protoc-3.6.1-win32.zip。ProtoBuf 开源有两个版本 proto2 和 proto3，他们之间并不是完全兼容的，本篇文章都是以 proto3 为例。

使用

使用 ProtoBuf 首先需要编写 proto 文件，在 proto 文件中定义数据结构，然后使用 protoc 编译 proto文件，protoc 会根据输入参数编译出对应语言的源文件，例如 .java 文件。

首先我们创建一个 demo.proto 文件，里面定义了一个数据结构 SearchRequest.

在 proto 中，所有结构化的数据都被称为 message。 message 里面定义了具体的字段。每个字段由修饰符、字段类型、字段名、字段编号组成，ProtoBuf 是多语言支持的，它定义了一套字段类型，在编译的时候再转化成目标语言的类型，具体的对应关系可以查看官方文档。

有以下几种修饰符

修饰符 说明
required 该字段为必要字段
optional 该字段为可选字段、默认是 optional
repeat 表示该字段可重复，转化成 java 代码的时候会被转成 List

其他暂且不理，我们先来编译看看

-I=$SRC_DIR : 指定输入目录

–java_out=$DST_DIR : 指定输出 java 文件的目录

protoFile : 要编译的proto文件

编译成功后会在指定的输出目录生成一个 java 文件，默认生成的类名是 proto 文件的文件名，在 proto 文件中定义的 message 会转成一个静态内部类。由于生成的类行数太多，就不贴出来了，自行编译就可以看到。

可以通过参数设置生成的类名和包结构

命名规范

Google 在官方文档中对 proto 提出了命名的规范，因为 proto 文件会转成很多语言，这些语言的通用命名规范互不相同。

  • message 和 enum 使用驼峰命名，并且首字符大写
  • message 中的字段名使用小写，用下划线分隔
  • enum 的值全部大写，用下划线分隔

更新

随着版本的迭代，我们可能需要修改之前定义的 proto 文件，这分两种情况，增加字段和删减字段

增加字段

ProtoBuf 是通过字段编号来解析的，你需要增加字段的时候，只需要在之前定义的 message 中直接新增字段就可以了

删减字段

如果在新版本中有旧的字段废弃了，有以下几种办法：

  • 保留该字段，但是不再使用，可以对旧字段重命名，例如增加 OBSOLETE_ 前缀
  • 删除该字段，但是需要使用reserved声明该字段编号保留，防止以后误用该字段编号
